## Summary

The long-path hardening on `main` fixed core file I/O, but `ShellDeleteFile()` still falls
back to a shell delete path that is not long-path safe when "remove to recycle bin" is
enabled.

Current behavior in `srchybrid/OtherFunctions.cpp`:

- existence is checked with `LongPathSeams::PathExists()`
- recycle-bin mode copies the path into a fixed `TCHAR todel[MAX_PATH + 1]`
- deletion is performed via `SHFileOperation`
- non-recycle delete uses `LongPathSeams::DeleteFile()`

That leaves an inconsistent boundary: deep paths can be opened, saved, moved, and deleted
directly, but delete-to-recycle-bin remains capped by the shell path/buffer model.

## Why This Matters

This is not theoretical. `ShellDeleteFile()` is used from live UI delete flows including:

- `srchybrid/DownloadListCtrl.cpp` for part-file removal
- `srchybrid/SharedFilesCtrl.cpp`
- `srchybrid/SharedDirsTreeCtrl.cpp`
- `srchybrid/CollectionCreateDialog.cpp`

For deep download/share paths this can produce failed deletes, truncated paths, or shell
behavior that diverges from the long-path-safe core filesystem layer.

## Evidence

### Local code

- `srchybrid/OtherFunctions.cpp`
  - fixed-size `MAX_PATH + 1` recycle-bin buffer
  - `SHFileOperation` recycle-bin delete path
- call sites:
  - `srchybrid/DownloadListCtrl.cpp`
  - `srchybrid/SharedFilesCtrl.cpp`
  - `srchybrid/SharedDirsTreeCtrl.cpp`
  - `srchybrid/CollectionCreateDialog.cpp`

### External reference

Microsoft documents that:

- `SHFILEOPSTRUCT`/`SHFileOperation` is superseded by `IFileOperation`
- source paths should be fully qualified
- `SHFileOperation` fails on `\\?\\` paths

Reference:
- <https://learn.microsoft.com/en-us/windows/win32/api/shellapi/ns-shellapi-shfileopstructa>

## Recommended Fix Direction

- Keep `LongPathSeams` as the core file boundary.
- Split recycle-bin delete from raw file delete explicitly.
- For long paths, either:
  - use a documented modern shell path, or
  - fall back deterministically to direct delete with a clear/logged behavior contract.
- Do not keep the current `MAX_PATH` scratch buffer.

## Test Requirements

- add a real-FS regression covering delete of a file under a deep Unicode path
- cover both:
  - recycle-bin enabled path
  - recycle-bin disabled path
- include a UI-facing seam/integration check for part-file deletion initiated from the
  download list path
