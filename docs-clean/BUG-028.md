## Summary

Current `main` passes MP3 paths to `id3lib` through `CStringA` even though the
app tracks file paths as UTF-16.

The vendored `id3lib` still exposes only narrow filename APIs and still opens
or updates files through narrow CRT / `CreateFileA` paths. On filenames that do
not round-trip through the active ANSI code page, MP3 tag parsing can fail or
silently miss metadata. The same integration also still depends on local
workarounds for broken `id3lib` Unicode getters and mixed ID3v1/ID3v2 Unicode
corruption.

## Evidence In Current Tree

- `srchybrid/FileInfoDialog.cpp:731-764`
  - local `ID3_GetStringW(...)` wrapper documents two live `id3lib` Unicode
    bugs:
  - only trust `id3lib` Unicode when the frame is already UTF-16
  - do not use `ID3_FieldImpl::Get(unicode_t*, ..., itemNum)` because
    `GetRawUnicodeTextItem` is broken
- `srchybrid/FileInfoDialog.cpp:883-888`
  - current `main` does:
  - `CStringA strFilePathA(pFile->GetFilePath());`
  - `myTag.Link(strFilePathA, ID3TT_ID3V2);`
- `srchybrid/KnownFile.cpp:1416-1426`
  - same ANSI path downcast and `myTag.Link(strFilePathA, ...)` pattern
- `repos\third_party\eMule-id3lib\include\id3\tag.h:99,134`
  - public API only exposes `Link(const char *fileInfo, ...)`
- `repos\third_party\eMule-id3lib\src\tag_impl.h:118,153,179`
  - implementation also only exposes `Link(const char*)`
  - `_file_name` is stored as narrow `dami::String`
- `repos\third_party\eMule-id3lib\src\utils.cpp:330-451`
  - file helpers still call `ifstream` / `fstream::open(name.c_str(), ...)`
- `repos\third_party\eMule-id3lib\src\tag_file.cpp:49-64,132-141,320-321`
  - truncate path still uses `CreateFileA`
  - file-link state is still stored as narrow `_file_name`
  - replacement path still uses narrow `remove(...)` / `rename(...)`
- `repos\third_party\eMule-id3lib\src\field_binary.cpp:128-163`
  - binary file import/export still uses narrow `fopen_s`

## Why This Is A Real Bug

`CStringA` conversion is locale-dependent and lossy for characters outside the
active ANSI code page. That means the MP3 file selected by the user can become
unopenable to `id3lib` even though the rest of eMule can still access it via
UTF-16 Win32 paths.

The current integration already acknowledges that parts of `id3lib`'s Unicode
path are broken. The app-side `ID3_GetStringW` wrapper and the forced
ID3v2-first read path are not speculative cleanup notes; they are live
workarounds for current dependency bugs.

## User-Visible / Runtime Impact

- MP3 metadata display can fail for files stored under Unicode-only filenames or
  directories
- media-tag extraction behavior varies by system locale / code page
- multi-item Unicode fields and mixed ID3v1 + ID3v2 files still rely on brittle
  defensive paths in app code

## Likely Fix Shape

1. Minimal current-main fix for the existing read-only usage:
   - stop passing `CStringA` paths from `FileInfoDialog.cpp` and `KnownFile.cpp`
   - open the file with the existing UTF-16 / long-path seam
   - feed bytes through `ID3_Tag::Link(ID3_Reader&)`, either with a custom
     reader over a wide-opened handle or by loading bytes into
     `ID3_MemoryReader`
2. Full vendor fix if `id3lib` remains strategic:
   - add wide or explicit UTF-8-normalized filename support to `ID3_Tag` and
     `ID3_TagImpl`
   - replace narrow helpers in `utils.cpp`, `tag_file.cpp`, and
     `field_binary.cpp`
   - revalidate `Update()` / `Strip()` temp-file replace logic on Win32 and
     long paths
   - fix or re-verify `ID3_FieldImpl::Get(unicode_t*, ...)` /
     `GetRawUnicodeTextItem`
3. Alternative dependency decision:
   - remove or replace `id3lib`, as the sibling trees already chose to do

## Validation Target

- place an MP3 under a path containing characters outside the active ANSI code
  page, ideally also under a long path
- verify File Info and metadata extraction still populate title / artist / album
- include a mixed ID3v1 + ID3v2 sample and a multi-item Unicode text-frame sample
