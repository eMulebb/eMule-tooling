## Summary

Client UDP send encryption is now gated by `thePrefs.IsCryptLayerEnabled()`.
That may be intentional, but it is protocol-adjacent and can affect Kad,
LowID, buddy, callback, or peer UDP interoperability when global crypt settings
are disabled.

This blocks R1 until the behavior is either proven compatible or adjusted with a
narrow documented exception.

## Evidence

- `srchybrid/ClientUDPSocket.cpp:449` gates outgoing UDP encryption overhead on
  `thePrefs.IsCryptLayerEnabled()`.
- `srchybrid/ClientUDPSocket.cpp:513` stores `newpending->bEncrypt` only when
  the global crypt layer is enabled.
- The branch diff from `release/v0.72a-community` shows this as a behavior
  change from the older Kad/key condition.

## Execution Plan

1. Revalidate the intent of `BUG-016` and the current client UDP encryption
   gating against legacy eMule behavior.
2. Build a focused regression matrix:
   - crypt enabled, ED2K UDP peer packet
   - crypt enabled, Kad UDP packet
   - crypt disabled, ED2K UDP peer packet
   - crypt disabled, Kad UDP packet
   - LowID callback/buddy path where available
3. Compare against `release/v0.72a-community` where the existing harness can
   produce meaningful parity evidence.
4. If compatibility is broken, restore the previous Kad-specific behavior with
   a narrow code comment and regression test.
5. If compatibility is not broken, record the proof in this item and the R1
   execution plan.

## Acceptance Criteria

- R1 has explicit evidence for Kad/client UDP behavior with crypt enabled and
  disabled.
- Any intentional behavior change is documented and covered by tests.
- LowID and callback paths are not regressed.

## Validation

- `pwsh -File repos\eMule-build\workspace.ps1 validate`
- Targeted Kad/client UDP regression tests or live-diff proof.
- Community-baseline comparison where supported by the harness.
